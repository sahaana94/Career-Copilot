# Interview Prep

The goal is a cheat sheet the user can skim in the 30 minutes before an interview — concrete, specific to this role and company, and built so they walk in with real stories ready rather than generic advice.

## 1. Gather context

Ask for whatever isn't already clear from the conversation:
- **Company and role** — and the job description, if it hasn't already been shared.
- **Interview stage/type**, if known — recruiter screen, hiring manager, panel, technical, executive — since the right prep differs by stage (a recruiter screen is more about narrative and logistics; a panel with the hiring manager goes deeper on role-specific scenarios).
- **Who they're meeting with**, if known (names/titles) — useful for tailoring questions and for the "questions to ask" section.

## 2. Company and role context

If web search or browsing tools are available, do a quick check for anything timely and relevant: recent news, product launches, funding, or strategic shifts at the company — things that could come up ("What do you know about us?") or that are genuinely useful for the user to know. Keep this brief and factual; don't pad the cheat sheet with generic "about the company" boilerplate the user could read on the website themselves.

If no browsing is available, work from the job description and whatever the user already knows about the company.

## 3. Map the JD to the candidate's real experience

This is the core of the cheat sheet. Take the top 4–6 requirements or responsibilities from the JD and, for each one, pull the most relevant story from `Career Profile/profile.md`'s Work History. Frame each as a ready-to-tell story — situation/task, what they did, the result (ideally quantified). The point is that the user should be able to glance at "they'll probably ask about X" and immediately recall *which* story to tell, not have to construct one on the spot.

If a fit assessment was already produced for this role, its **Top Strengths** are a good starting point for which stories to feature.

## 4. Likely questions

Generate a mix of:
- **Behavioral questions** likely for this stage/seniority (e.g., "Tell me about a time you had to influence without authority," "Describe a project that didn't go as planned").
- **Role-specific questions** drawn directly from the JD's responsibilities and the company's likely priorities.

For each question, give a short prep note — not a full scripted answer, but enough to jog the right memory and structure (e.g., "→ the GeneXpert launch story: emphasize the cross-functional scope and the regulatory timeline").

## 5. Honest gaps — and how to handle them if asked

Pull from `Career Profile/profile.md`'s Known Gaps (and the fit assessment's Honest Gaps, if one exists for this role) anything that might come up directly ("I see you haven't worked in X — how would you approach that?"). For each, suggest a brief, honest way to address it — acknowledging the gap, and pointing to the closest adjacent experience or how they'd ramp up. The goal is to help the user not get caught off guard, not to help them dodge the question.

## 6. Questions to ask the interviewer

3–5 questions tailored to the role, company, and (if known) the specific interviewer — substantive questions that show genuine engagement, not generic ones ("What's the culture like?") that could apply to any company.

## 7. Output format

Produce a structured document (markdown or `.docx`, matching whatever format the user's other prep materials use) with these sections:

```markdown
# Interview Prep — [Company] — [Role]

## Company & Role Snapshot
[brief, factual]

## Your Strongest Stories for This Role
- [Requirement/theme] → [story from Work History, with the result]
- ...

## Likely Questions & Prep Notes
- [Question] → [prep note: which story / angle to use]
- ...

## If They Ask About: [Gap]
[honest framing + adjacent experience to point to]

## Questions to Ask Them
- ...
```

Save it alongside other materials for this application (e.g., the same folder as the tailored resume), named something like `[Name]_[Company]_Interview_Prep`.

## Tone

Same honesty principle as the rest of this skill: prep should make the user more confident *because* it's accurate, not because it papers over real gaps. A user who walks in over-prepared for questions about a gap they actually have will do better than one who hoped it wouldn't come up.
